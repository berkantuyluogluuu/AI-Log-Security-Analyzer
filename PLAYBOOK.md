# INCIDENT RESPONSE PLAYBOOK: SQL INJECTION (SQLi) ATTACK

**Document ID:** IR-PB-SQLI-004  
**Classification:** Internal Use Only  
**Target Audience:** SOC Analysts, Incident Response (IR) Team, Database Administrators (DBAs), Application Security Engineers  
**Framework Alignment:** NIST SP 800-61 Rev. 2 & ISO/IEC 27001:2022  

---

## 1. DETECTION & ANALYSIS

The primary objective of this phase is to identify, validate, and assess the scope of a suspected SQL Injection attempt or successful breach.

### 1.1 Alerts and Indicators of Compromise (IoCs)
* **WAF Alerts:** High-frequency triggers on SQL syntax keywords (`UNION SELECT`, `OR 1=1`, `WAITFOR DELAY`, `--`).
* **Database Logs:** Unusually high volumes of database errors (e.g., syntax errors, permission denied) or unauthorized schema modifications.
* **Application Logs:** HTTP 500 Internal Server Errors originating from input fields, search bars, or URL parameters.

### 1.2 Triage & Verification (In-House Tool Integration)
To filter out false positives and confirm active exploitation, the SOC team must immediately utilize the proprietary **Python Log Analyzer Tool**.

* **Step 1:** Execute the internal **Python Log Analyzer** against the web server access logs (Nginx, Apache, or IIS) and database query logs for the suspected timeframe.
* **Step 2:** Review the structured output (JSON/CSV) generated by the analyzer to isolate:
    * The attacker's source IP addresses and User-Agent strings.
    * The specific vulnerable URI endpoints and injected payloads.
    * The success status of the injection (e.g., matching HTTP 200 responses with anomalous payload lengths vs. HTTP 403/500).
* **Step 3:** Cross-reference the Python tool's findings with SIEM telemetry to determine if data exfiltration has occurred (e.g., outbound data spikes).

### 1.3 Severity Classification
Based on the log analysis, classify the incident:
* **LOW:** Blocked probes/scanning with no successful execution.
* **MEDIUM:** Successful injection exposing non-sensitive schema info.
* **HIGH:** Confirmed data exfiltration of PII/credentials or unauthorized administrative access achieved via SQLi.

---

## 2. CONTAINMENT

Once a live SQLi attack or breach is confirmed, immediate containment actions must be taken to limit the blast radius and prevent further data loss.

### 2.1 Short-Term Isolation
* **Network Level:** Apply temporary blocks on the perimeter Firewall/WAF for the malicious source IPs identified by the Python log analyzer.
* **Application Level:** If a specific endpoint is targeted and vulnerable, deploy an emergency WAF virtual patch targeting that specific URI parameter. If virtual patching is unavailable, temporarily disable the affected application feature/form.
* **Database Level:** Revoke excessive privileges or temporarily disable the compromised database user account driving the web application connections.

### 2.2 System Isolation (High Severity Only)
* If the attacker has gained Command Execution (`xp_cmdshell` or similar OS-level access via database), isolate the database server and associated web servers from the network entirely by shifting them to a quarantine VLAN.
* **CRITICAL:** Do not power off the machines to preserve volatile memory (RAM) for forensic investigation.

---

## 3. ERADICATION & RECOVERY

Eradication aims to eliminate the root cause of the vulnerability, while Recovery ensures systems are safely returned to a trusted production state.

### 3.1 Eradication (Vulnerability Remediation)
* **Code Fixes:** Coordinate with the Development team to replace all dynamic, concatenated SQL queries in the source code with **Parameterized Queries (Prepared Statements)** or utilize a secure Object-Relational Mapper (ORM).
* **Input Validation:** Implement strict allow-list input validation on both client and server sides.
* **Database Hardening:**
    * Disable dangerous extended stored procedures (e.g., `xp_cmdshell` in MS SQL, `LOAD_FILE()` in MySQL).
    * Enforce the Principle of Least Privilege: Ensure the web application's database user only has `SELECT`, `INSERT`, and `UPDATE` permissions on required tables—strictly no `DROP`, `ALTER`, or `DB_OWNER` privileges.

### 3.2 Recovery & Verification
* **Data Integrity Check:** Run database consistency checks (DBCC) and compare cryptographic checksums against pre-incident backups to ensure data has not been maliciously altered.
* **Patch Deployment:** Push the remediated code through the CI/CD pipeline after successful static/dynamic application security testing (SAST/DAST).
* **Relaunch:** Restore network connectivity and monitor database transactions closely using real-time WAF logging and the Python log analyzer to ensure no recurring anomalous activity occurs.

---

## 4. POST-INCIDENT ACTIVITY

This phase focuses on continuous improvement, documentation, and aligning systemic fixes with regulatory and compliance frameworks.

### 4.1 Lessons Learned
* Conduct a post-mortem meeting within 72 hours with the IR team, developers, and management.
* **Key Questions:** How did the vulnerability bypass initial code reviews? Did the Python Log Analyzer detect the threat fast enough? How can we optimize the analyzer's heuristics for future variants?
* Document the timeline, financial impact, and remediation steps in the central Incident Management System.

### 4.2 ISO/IEC 27001:2022 Control Alignment
To prevent recurrence and maintain standard compliance, the following ISO 27001 controls must be updated and audited based on this incident:

| ISO 27001 Control | Action Item Required |
| :--- | :--- |
| **A.8.19 (Secure coding)** | Update the organization's Secure Coding Guidelines to mandate parameterized queries. Integrate mandatory SAST scanning into the CI/CD pipeline to catch SQLi before deployment. |
| **A.8.20 (Network security)** | Tune WAF inspection rules specifically for SQL injection vectors and signatures. |
| **A.5.7 (Threat intelligence)** | Feed the malicious IPs and indicators identified by the Python tool into the corporate Threat Intelligence Platform (TIP). |
| **A.8.24 (Use of cryptography)** | Review encryption status of sensitive data at rest within the database. If an attacker accesses the tables via SQLi in the future, critical data (like passwords or PII) must remain strongly hashed/encrypted. |
